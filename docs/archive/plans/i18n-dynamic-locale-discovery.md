# i18n 動的ロケール対応プラン

## Context

現在 i18n のロケール一覧が `src/i18n/routing.ts` に `["ja", "en"]` とハードコーディングされている。今後ロケールを追加する際、新しい `messages/{locale}/` ディレクトリを作るだけで自動的に認識されるようにしたい。不完全な翻訳ファイルは `en` にフォールバックする。

**重要な制約**: `routing.ts` は `"use client"` コンポーネント (`language-switcher.tsx`) からインポートされるため、`fs` モジュールを直接使えない。ビルド時コード生成で解決する。

**破壊的変更**: `defaultLocale` を `"ja"` → `"en"` に変更する（ユーザー判断済み）。これにより proxy リダイレクト先、`Accept-Language` 未対応時のフォールバック先、SEO インデックスが変わる。

## 変更概要

### 1. ロケール検出スクリプト (新規: `scripts/discover-locales.ts`)

`messages/` ディレクトリをスキャンし、`src/i18n/discovered-locales.ts` を生成する。

```typescript
// scripts/discover-locales.ts
import fs from "node:fs";
import path from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const MESSAGES_DIR = path.resolve(__dirname, "../messages");
const OUTPUT = path.resolve(__dirname, "../src/i18n/discovered-locales.ts");
const DEFAULT_LOCALE = "en";

// BCP 47 ロケールコードのバリデーション
const LOCALE_PATTERN = /^[a-z]{2,3}(-[A-Za-z0-9]{2,8})*$/;

const locales = fs.readdirSync(MESSAGES_DIR, { withFileTypes: true })
  .filter(e => e.isDirectory())
  .filter(dir => LOCALE_PATTERN.test(dir.name))
  .filter(dir => {
    const files = fs.readdirSync(path.join(MESSAGES_DIR, dir.name));
    return files.some(f => f.endsWith(".json"));
  })
  .map(e => e.name)
  .sort();

if (locales.length === 0) {
  console.error("[i18n] No valid locale directories found in messages/");
  process.exit(1);
}

const safeDefault = locales.includes(DEFAULT_LOCALE) ? DEFAULT_LOCALE : locales[0];

const code = `// AUTO-GENERATED by scripts/discover-locales.ts — DO NOT EDIT
export const discoveredLocales = ${JSON.stringify(locales)} as const;
export const defaultLocale = ${JSON.stringify(safeDefault)} as const;
`;

// 内容が同一ならスキップ（vitest --watch の無限ループ防止）
const existing = fs.existsSync(OUTPUT) ? fs.readFileSync(OUTPUT, "utf-8") : "";
if (existing === code) {
  console.log("[i18n] discovered-locales.ts is up to date");
  process.exit(0);
}

fs.writeFileSync(OUTPUT, code, "utf-8");
console.log(`[i18n] Discovered locales: ${locales.join(", ")} (default: ${safeDefault})`);
```

設計ポイント:

- `fileURLToPath` + `path.dirname` で Node.js 20 互換
- BCP 47 正規表現でディレクトリ名をバリデーション（`.DS_Store`, `__proto__` 等を除外）
- `defaultLocale` 出力に `JSON.stringify` を使用（テンプレートリテラル注入を防止）
- ロケール 0 件時にエラー終了（サイレント失敗を防止）
- 生成内容が既存ファイルと同一の場合はスキップ（`vitest --watch` 無限ループ防止）

生成ファイルの管理方針:

- **バージョン管理に含める**（`.gitignore` に追加しない）
- CI で `npm run i18n:discover && git diff --exit-code src/i18n/discovered-locales.ts` を実行し乖離を検出
- npm lifecycle hooks でも自動実行: `"predev"`, `"prebuild"`, `"pretest"`

### 3. `src/i18n/messages.ts` — `fs.readFileSync` ベースのフォールバック付き読み込み

**方針変更（レビュー反映）**: `import()` ではなく `fs.readFileSync` + `JSON.parse` に切り替え。

理由: 部分翻訳ロケールで存在しない JSON ファイルを `import()` するとバンドラー（Webpack/Turbopack）がモジュール解決時にエラーを出す。`try/catch` では捕捉できない。`loadAllMessages` / `loadNamespaces` は `getRequestConfig` (サーバーサイド) 内でのみ呼ばれるため `fs` の使用が可能。

実装内容:

- `fs.readFileSync` + `JSON.parse` でネームスペースファイルを読み込み
- `deepMerge()` ユーティリティを追加（プロトタイプ汚染対策付き）
- non-default locale は en をベースにして deep-merge
- 不在のネームスペースファイルはファイル存在チェック後に空オブジェクトにフォールバック（warning ログ出力）

```typescript
const DANGEROUS_KEYS = new Set(["__proto__", "constructor", "prototype"]);

function deepMerge(
  base: Record<string, unknown>,
  overlay: Record<string, unknown>,
): Record<string, unknown> {
  const result = { ...base };
  for (const [key, value] of Object.entries(overlay)) {
    if (DANGEROUS_KEYS.has(key)) continue;
    if (
      value !== null && typeof value === "object" && !Array.isArray(value) &&
      typeof result[key] === "object" && result[key] !== null && !Array.isArray(result[key])
    ) {
      result[key] = deepMerge(result[key] as Record<string, unknown>, value as Record<string, unknown>);
    } else {
      result[key] = value;
    }
  }
  return result;
}
```

### 4. `src/components/layout/language-switcher.tsx` — `Intl.DisplayNames` 使用

`t(loc)` (JSON キーベース) → `Intl.DisplayNames` API に切り替え。ロケール追加時に LanguageSwitcher.json の更新が不要になる。

**表記方式**: 自国語表記（endonym）を採用。1Password/Bitwarden と同様、ユーザーが自分の言語をその言語自体の表記で探せるようにする。

```typescript
function getLocaleDisplayName(localeCode: string): string {
  try {
    // 自国語表記: "ja" → "日本語", "en" → "English", "fr" → "français"
    return new Intl.DisplayNames([localeCode], { type: "language" }).of(localeCode) ?? localeCode;
  } catch {
    return localeCode;
  }
}
```

### 7. `src/i18n/messages-consistency.test.ts` — 部分翻訳を許容 + セキュリティクリティカル NS

- default locale (en) は全ネームスペース・全キー必須（厳密チェック）
- non-default locale は部分翻訳を許容（余分なキーがないことのみ検証）
- **セキュリティクリティカルネームスペース**は全ロケールで完全翻訳を必須とする

```typescript
const SECURITY_CRITICAL_NAMESPACES: readonly string[] = [
  "Vault", "Recovery", "VaultReset", "RepromptDialog", "Auth", "EmergencyAccess",
];
```

具体的なテスト分割:

- 既存の「keeps key sets aligned between locales per namespace」テストを2つに分割:
  - **security critical NS**: 全ロケールでキー完全一致を要求（現行通り）
  - **その他の NS**: non-default locale でキー不足は許容、ただし en に存在しないキーがあれば fail（孤立キー検出）
- `loadLocale()` を修正: ファイル不在時は空オブジェクト `{}` を返す（default locale を除く）

## 新規テスト計画

### T-A: `scripts/__tests__/discover-locales.test.ts`

- 正常系: `messages/en/`, `messages/ja/` からロケール配列が生成される
- `.DS_Store`、`README.md` 等がロケールとして検出されない
- BCP 47 に準拠しないディレクトリ名が除外される
- 空ディレクトリ（JSON なし）が除外される
- `messages/` ディレクトリが存在しない場合のエラーハンドリング
- 生成内容が同一の場合にスキップされる

### T-B: `src/i18n/messages.test.ts` — `deepMerge()` + フォールバック検証

- `deepMerge` の基本動作: 第2引数が優先
- `deepMerge` のフォールバック: 不足キーは第1引数（base）が残る
- `deepMerge` のプロトタイプ汚染対策: `__proto__` キーが除外される
- 配列値: 上書き（マージではない）
- `loadAllMessages` のフォールバック: fixture ロケールで不足 NS の値が en と一致
- フォールバック発生時に warning ログが出力される

### T-C: `src/lib/email/templates/emergency-access.test.ts` — getLabels フォールバック

- `emergencyInviteEmail("fr", "Alice")` が en の subject/html/text を返す（en フォールバック）
- `emergencyInviteEmail("ja", "Alice")` が ja の内容を返す（回帰テスト）
- `emergencyInviteEmail("", "Alice")` が en にフォールバックすること

### T-D: テストの決定性

ロケール一覧に依存するテストは `vi.mock("@/i18n/discovered-locales")` でモックし、ローカルの `messages/` 構成に依存しない決定的なテストとする。

## 型安全性

`AppLocale` 型は `(typeof routing.locales)[number]` で導出。生成ファイルが `as const` を使うため、リテラル型 (`"en" | "ja"`) が維持される。新しいロケール追加時は生成ファイル再生成で型も更新される。
