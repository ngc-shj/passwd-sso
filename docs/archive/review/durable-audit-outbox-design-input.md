# Proposal 2: Durable Audit Backend (Transactional Outbox) — Design Input

This file is the input for the `/multi-agent-review` skill when starting a new
session for Proposal 2. It is not a plan — the plan will be generated by the
review skill from this input.

---

## Background

Today, audit logging in passwd-sso has in-memory retry only — if the process
crashes, buffered events are lost. The `logAudit` function writes directly
to the `audit_logs` table and best-effort retries via `audit-retry.ts`.

The goal is to make audit logging **durable** so that:
- Business success + audit failure cannot happen
- Process restart does not lose audit events
- Retry is reliable, observable, and bounded
- Optional: tamper-evidence for compliance

## Proposed evolution (4 stages)

### Stage 1: Outbox table

Audit events are written to a new `audit_outbox` table in the **same DB
transaction** as the business logic. A separate worker reads `pending` rows
and forwards them to the final destination (DB audit table, SIEM, webhook).

```
audit_outbox
  id
  tenant_id
  actor_id
  event_type
  payload_json
  status (pending/processing/sent/failed)
  attempt_count
  created_at
  next_retry_at
  last_error
```

Properties:
- The business write and the audit outbox write are atomic — either both
  commit or both roll back. No silent audit loss.
- Worker can be a separate process or a cron job.
- Failed deliveries can be retried with backoff.
- Dead-letter (`status=failed` after max retries) is queryable.

### Stage 2: Worker / queue consumer

A background process polls `pending` rows, marks them `processing`, attempts
delivery, then transitions to `sent` / `failed`. Concurrency, backoff, and
dead-letter handling are explicit.

### Stage 3: Multiple delivery targets

The worker can fan out to multiple destinations:
- DB `audit_logs` table (existing)
- SIEM (Splunk, Sentry, Datadog, etc.)
- Object storage (S3, GCS) for long-term retention
- Webhook (already exists)

Per-target failure isolation: if SIEM is down, DB writes still succeed.

### Stage 4: Tamper-evidence (optional)

For high-assurance compliance:
- `prev_hash` + `event_hash` per row → hash chain
- Append-only / WORM-like retention
- External notarization (timestamping authority)

## Operational metrics

The worker / health endpoint should expose:
- pending count
- oldest pending age
- failed count
- dead-letter count
- average retry count

## Initial threat model

- **Worker compromise** → minimum-privilege DB role for the worker (read
  outbox, write audit_logs only)
- **SSRF in delivery target URL** → validate against allowlist, share
  webhook validator
- **Sensitive metadata leakage** → `sanitizeMetadata()` (existing) must
  apply to outbox writes too
- **Hash chain race** → `SELECT FOR UPDATE` or sequence-number ordering
- **Stuck `processing` rows** → reaper that resets to `pending` after timeout

## Out of scope considerations

- The existing `audit-retry.ts` in-memory buffer should be deprecated in
  favor of the outbox; its behavior overlaps and could double-write.
- Cross-actor audit (`actorType` enum) is already in place — outbox is
  agnostic to actor type.

## Phase plan suggestion

| Phase | Scope |
|---|---|
| 1 | Outbox table + atomic write helper, worker + DB target only |
| 2 | Operational metrics + reaper |
| 3 | Multiple delivery targets (SIEM, S3) |
| 4 | Hash chain (optional, only if compliance demands it) |

## Reference: prior multi-agent review findings on Proposal 2

(From the original `/multi-agent-review` run on the bridge code + outbox
proposal pair, before either was implemented. Numbering: P2-S* are security
findings, P2-T* are testing findings, X-M1 is cross-cutting.)

- **P2-S1 (Major)**: Worker `withBypassRls` purpose classification not
  specified — must use `BYPASS_PURPOSE.AUDIT_WRITE` explicitly.
- **P2-S2 (Major)**: Worker process minimum privilege not designed —
  use a separate DB role analogous to `passwd_app` / `passwd_user`.
- **P2-S3 (Major)**: Stage 3 dynamic delivery target URL → SSRF risk.
  Reuse webhook validator + private IP range deny.
- **P2-S4 (Major)**: Stage 4 hash chain concurrent computation has no
  ordering guarantee — `SELECT FOR UPDATE` or sequence number required.
- **P2-S5 (Minor)**: `sanitizeMetadata()` not specified for outbox writes.
- **P2-S6 (Minor)**: `processing` stuck reaper design not defined —
  add `processing_started_at` + timeout reaper.
- **P2-T1 (Critical)**: "Outbox + business in same transaction" invariant
  is the most important property and there is no test design for it.
- **P2-T2 (Critical)**: Worker state machine (`pending → processing → sent
  / failed → retry / dead-letter`) has no test plan.
- **P2-T3 (Major)**: Existing test fixtures (`FULL_POLICY_RESPONSE`,
  `BASE_POLICY`) need new field updates.
- **P2-T4 (Major)**: `account-lockout.ts` test does not yet support
  configurable thresholds.
- **P2-T5 (Major)**: Stage 3 multi-target routing not designed for
  testability — extract a pure function.
- **P2-T6 (Major)**: `auditLogRetentionDays` enforcement test missing.
- **P2-T7 (Minor)**: Operational metrics endpoint test not planned.
- **P2-T8 (Minor) [Adjacent]**: `status` enum not declared as a Prisma
  enum → arbitrary string risk.
- **X-M1 (Cross-cutting)**: Atomic consume (Proposal 1) and outbox
  transaction boundary (Proposal 2) cannot be verified by mocked unit
  tests alone (project's known integration test gap).

A new `/multi-agent-review` run will likely re-derive most of these.
