### S5 — Cryptographic version metadata integrity (C7)

1. As a team writer with an existing entry (itemKeyVersion=1, aadVersion=2), POST `PUT /api/teams/<teamId>/passwords/<entryId>` with body `{ itemKeyVersion: 2 }` and NO `encryptedBlob`/`encryptedOverview` → expect 409 `KEY_VERSION_WITHOUT_REENCRYPT`.
2. Same call with `{ encryptedBlob, encryptedOverview, itemKeyVersion: 2, teamKeyVersion: 1 }` → expect 200.
3. As personal vault user, attempt `PATCH /api/passwords/<id>` with `{ aadVersion: 3 }` and no `encryptedBlob` → expect 409.
4. No-op write: `PATCH /api/passwords/<id>` with `{ aadVersion: <existing value> }` no blob → expect 200 (positive path).
5. Send `itemKeyVersion: 999999999999` (> 2^31-1) → expect 400 from Zod max bound (Round 1 S-7 / RS3).

## Adversarial scenarios (Tier-2 required)

### A1 — Cross-tenant SA token reuse
Attempt to use a Tenant-1 SA token against Tenant-2's `/api/vault/delegation/check`. Expect 403 (existing `enforceAccessRestriction`).

### A2 — Token-as-step-up bypass
Forge a "recent step-up" claim in a Bearer token (e.g., set a payload field that LOOKS like the recent-auth marker). Verify `requireRecentCurrentAuthMethod` reads from session cookie ONLY (`getSessionToken(req)`) and the token claim is ignored. Step-up cannot be satisfied without a real session.

### A3 — Scope elevation via DCR
Register a new MCP client via DCR. Attempt to call `delegation/check` with the client's default issued token immediately (before any user consent). Verify the token does NOT carry `delegation:check` scope by default (only after explicit consent grant). Expect 403.

### A4 — Body size content-length spoofing + chunked-TE bypass
Two sub-scenarios:

- **(a) Content-Length spoofing**: Send `Content-Length: 100` in the header but actual body of 5 MB. With `parseBody`'s stream-byte-counting (Round 2 S-2R), the server should terminate the request after `total > maxBytes` (~1 MB or per-route override). The header value is ignored once the stream byte count exceeds the cap.
- **(b) Chunked-TE no-Content-Length**: Send `Transfer-Encoding: chunked` with no `Content-Length` header, body 5 MB. `parseBody`'s stream-counting fires the same way (`req.body.getReader()` reads chunks; total accumulates; abort at cap).

Note (Round 2 S-2R correction): Round 1 proposed a `next.config.ts` platform backstop. That approach was incorrect — Next.js 16 App Router has NO platform body cap for route handlers (`serverActions.bodySizeLimit` is for Server Actions only, `proxyClientMaxBodySize` is for middleware proxy stream cloning). The stream-byte-counting inside `parseBody` is the authoritative guard. Verify by inspecting `parse-body.ts` reads `req.body` as a stream, not via `req.json()`.

### A5 — SA token JIT-requesting unsupported scope
After this PR ships, attempt to create an access-request requesting `delegation:check` scope via `POST /api/tenant/access-requests` as SA. Expect: if `SA_TOKEN_SCOPE.DELEGATION_CHECK` is intentionally NOT added (per F-2 decision), this should fail validation (Zod enum reject) because the scope is not in `SA_TOKEN_SCOPES`.
