## Functionality Findings

- **F1 ‚Äî Critical**: `PasswordRow` / `EntryActionsMenu` (`password-row.tsx:42-88`, `entry-actions-menu.tsx:43-59`) expose NO restore / delete-permanently affordance (`rg restore|deletePermanently|permanent` ‚Üí 0 matches). Today's trash UIs (`trash-list.tsx:311-344`, `team-trash-list.tsx:380-414`) render bespoke `<Card>`s with explicit Restore + Delete-permanently buttons + per-row confirm dialogs. Trash 3-pane is therefore NOT "incidental" ‚Äî it requires net-new shared row affordances + destructive confirm dialogs. **Resolution: add contract C9 (trash row actions in `EntryActionsMenu`/`EntryListView`, gated by descriptor); add Batch 2.5.** ‚úÖ addressed.
- **F2 ‚Äî Critical**: trash overview type is a strict SUBSET of `PasswordRowEntry` (trash decrypts only `{id,entryType,title,username,snippet,brand,lastFour,fullName,idNumberLast4,deletedAt}` ‚Äî `trash-list.tsx:85-96`), not a superset. To satisfy `E extends PasswordRowEntry & PasswordDetailPaneEntry`, the trash adapter must decrypt the FULL overview (all per-type fields + tags). That is an intentional widening of in-memory decrypt + a richer trash row render. **Resolution: state widening explicitly in C5/C6; correct the "same fields as today" acceptance for trash; security note on residency.** ‚úÖ addressed.
- **F3 ‚Äî Major**: `ARCHIVE_VIEW.detailReadOnly === true` (INV-C2.2) contradicts current behavior ‚Äî archive IS editable today (`password-dashboard.tsx:406` editable; team `team-archived-list.tsx:508` `readOnly={!roleCanEdit}` + `onEdit`). Making archive read-only is a regression mislabeled as parity. **Resolution: `ARCHIVE_VIEW.detailReadOnly = false`; only TRASH_VIEW is read-only.** ‚úÖ addressed.
- **F4 ‚Äî Major**: empty-trash + trash row actions permission gating differs: personal shows empty-trash to all (owner-only vault); team gates empty-trash AND per-row restore/delete behind `OWNER|ADMIN` (`team-trash-list.tsx:277,380`); a team MEMBER sees trash as a read-only list with NO action buttons. Plan has no `canRestore`; mapping is underspecified. **Resolution: trash row actions + empty-trash gated by `canDelete` for team, unconditional for personal; documented in C6/C9.** ‚úÖ addressed.
- **F5 ‚Äî Major**: mutation methods must preserve optimistic list-removal, rollback-on-failure, AND `notify*DataChanged` sidebar-count event dispatch (team via `useTeamEntryMutations`; personal via `onDataChange`‚Üí`notifyVaultDataChanged`). C1's `Promise<void>` methods say nothing about these load-bearing side-effects. **Resolution: specify ownership in C1/C3 ‚Äî view does optimistic removal + rollback-reload; each mutation dispatches the data-changed event (explicit invariant INV-C3.x).** ‚úÖ addressed.
- **F6 ‚Äî Major**: per-entry decrypt-failure policy diverges ‚Äî personal SKIPS failed entries (`password-list.tsx:211`), team renders a `"(decryption failed)"` placeholder (`page.tsx:253`). One shared `fetchOverviewEntries` must preserve each vault's policy. **Resolution: note per-vault failure policy in C5/C6; `useEntryListData` imposes none.** ‚úÖ addressed.
- **F7 ‚Äî Minor**: INV-DEV1 (descriptor√óadapter capability assert) is not mechanically realizable from C1 ‚Äî no `supports` map; would require calling throwing methods or a parallel hardcoded map. **Resolution: add `supportsFavorite` (and capability flags as needed) to the adapter.** ‚úÖ addressed.
- **F8 ‚Äî Minor**: favorites view's "unfavorite removes row" optimistic behavior (`password-list.tsx:300-311`) is view√ómutation coupling not captured by "favorites reuse NORMAL_VIEW". **Resolution: add `removeOnUnfavorite` descriptor flag; favorites gets its own descriptor.** ‚úÖ addressed.
- **F9 ‚Äî Minor**: split-ownership hazard ‚Äî if `EntryListView` owns `activeEntry` but the dashboard still owns `selectionMode`, the view-change reset (clear active + clear selection, today atomic in one during-render block) desyncs. **Resolution: `EntryListView` owns BOTH activeEntry and selectionMode; parent passes `query`+`kind` (reset key) and hosts only dialogs.** ‚úÖ addressed.
- **F10 ‚Äî Minor**: selection-summary i18n key differs per vault (`selectedInPane` vs `selectedCount`). **Resolution: canonical key `selectedInPane` in `EntryListView`; verify ja/en.** ‚úÖ addressed.

## Security Findings

- **S1 ‚Äî Major**: no shared overview-decrypt helper. The blob path funnels through `mapDecryptedBlobToDetailFields`, but overview decrypt (separate `OVERVIEW` AAD scope, `crypto-aad.ts:50-54`) is relocated verbatim into each adapter with no consolidation ‚Äî recurring AAD-drift root cause ([[project_aad_distributed_contract_rootcause]]). **Resolution: mandate `decryptPersonalOverview` / `decryptTeamOverview` shared helpers (one AAD derivation each) symmetric to the blob mapper; add round-trip overview-AAD parity test; acceptance: no `OVERVIEW`-AAD literal outside the one helper.** ‚úÖ addressed.
- **S2 ‚Äî Major**: INV-S3 misdescribes the team gate. There is NO separate team gate ‚Äî both personal dashboard and team page render as children of `dashboard/layout.tsx:30`'s single `<VaultGate>` (`useVault().status`); team plaintext residency clears via that SAME personal-status unmount. Risk: an implementer "fixes" the non-existent team gate by hoisting the view above VaultGate, breaking clear-on-lock. **Resolution: correct INV-S3 to the actual mechanism + assert EntryListView stays a descendant of `dashboard/layout.tsx`'s VaultGate in BOTH mounts.** Pre-existing note: `clearAll` (`team-vault-core.tsx:154`) never called on lock ‚Äî cached team CryptoKeys are non-extractable, hold no plaintext, residency holds; tracked TODO, not a blocker. ‚úÖ addressed.
- **S3 ‚Äî Major**: `availability.ready=false` must ALSO gate the pane's `usePasswordEntryDetail`, not just the list. Today the team page passes `teamVaultStatus` (keyPending‚ÜíLOADING) into the hook (`page.tsx:504`). If `EntryListView` passes constant UNLOCKED while only hiding the list, a stale `activeEntry` + key-pending race fires `getEntryDecryptionKey` against an undistributed key. **Resolution: `EntryListView` derives the `vaultStatus` it passes to `usePasswordEntryDetail` FROM `adapter.availability` (`ready ? UNLOCKED : LOADING`); acceptance: not-ready ‚áí `buildGetDetail` never invoked even with non-null activeEntry.** ‚úÖ addressed.
- **S4 ‚Äî Minor**: preserve LAZY `buildGetDetail` invocation (`use-entry-actions.ts:90-91` builds closure per row; decrypt only on user callback). Risk: unified wiring eagerly resolves per-row ‚Üí multiple plaintexts resident (INV-S1 break). **Resolution: forbidden-pattern/acceptance ‚Äî `buildGetDetail(entry)()` invoked ONLY via active-pane hook + user-event callbacks; never per-row at render / hover / scroll. Port the hover-no-decrypt test.** ‚úÖ addressed.
- **S5 ‚Äî Minor**: plan says "GUEST"; the actual role is `VIEWER` (`team-role.ts:3-8`). **Resolution: replace GUEST‚ÜíVIEWER throughout; reproduce `page.tsx:303-306` permission derivation verbatim; acceptance: VIEWER adapter ‚áí canEdit=canDelete=canCreate=false, pane readOnly, no edit/share/delete affordance even if descriptor enables.** ‚úÖ addressed.
- **S6 ‚Äî Conditional/Minor**: keep `readOnly = descriptor.detailReadOnly || !permissions.canEdit` explicit; failure mode is dropping the descriptor term, letting OWNER reveal/edit a trashed entry. **Resolution: keep OR explicit; acceptance: TRASH_VIEW pane is readOnly for ALL roles incl. OWNER (reveal blocked, no edit button), independent of canEdit.** (Note: applies to TRASH only ‚Äî F3 establishes ARCHIVE is editable.) ‚úÖ addressed.

No Critical security findings. No new decrypt site re-derives AAD from scratch (adapters reuse `build*GetDetail` for blobs; S1 is a missed consolidation, not a new wrong-AAD derivation). Escalation: none flagged `escalate: true`.
