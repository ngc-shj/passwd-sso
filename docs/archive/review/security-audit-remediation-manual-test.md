# Manual Test: security-audit-remediation (R35 Tier-2)

Branch: `fix/security-audit-remediation`
Scope: Compose/infra surface + auth-flow UI surface. Pure API-level scenarios
already covered by unit/integration tests are excluded (two-filter rule).

---

## 1. Pre-conditions

### Required env vars

Add the following to `.env` (or run `npm run init:env`, which now prompts for both).
Values must be hex strings (URL-safe; `init:env` generates them automatically):

```
PASSWD_JACKSON_PASSWORD=<64-hex-char value, e.g. generated by: openssl rand -hex 32>
REDIS_PASSWORD=<64-hex-char value, e.g. generated by: openssl rand -hex 32>
```

Both live in the "External / Build-time" section at the bottom of `.env.example`,
alongside `PASSWD_OUTBOX_WORKER_PASSWORD`.

### Stack

Dev stack started via `npm run docker:up` (wraps
`docker compose -f docker-compose.yml -f docker-compose.override.yml up`).
All commands below assume the repo root as the working directory.

### Placeholders used throughout this document

| Placeholder | Meaning |
|---|---|
| `<test-user-email>` | A test account email address |
| `<second-user-email>` | A second distinct test account email address |
| `<host-port>` | The mapped host port (e.g. 5432, 6379) |
| `<sentinel-container>` | One of `sentinel-1`, `sentinel-2`, `sentinel-3` |
| `<client-name>` | An arbitrary DCR client display name |
| `<op-token>` | An operator API token minted from the dashboard |
| `<magic-link-url>` | The full magic-link URL received by `<test-user-email>` |

## 2. Steps / Expected Result

### Step 1 — Fresh boot health check

**Steps:**

1. Ensure both new env vars are present in `.env`.
2. `npm run docker:up` (or `docker compose -f docker-compose.yml -f docker-compose.override.yml up -d`).
3. Wait for all containers to report healthy (`docker compose ps`).
4. `curl -s http://127.0.0.1:3000/api/health/ready`
5. `docker compose logs jackson 2>&1 | grep -i "server listening\|started\|ready\|error"` (VE1: limited to boot + DB connectivity; real-IdP E2E not reproducible locally)
6. `redis-cli -h 127.0.0.1 -p 6379 ping`

**Expected:**

- Step 4: HTTP 200, body `{"status":"ok"}` (or equivalent ready envelope).
- Step 5: Jackson log shows a listening/started line; no DB connection error.
- Step 6: Response is `NOAUTH Authentication required.` — unauthenticated probe is rejected.

### Step 2 — Existing-volume Jackson role migration

> **Context:** `initdb/` scripts run only on first volume creation. Existing dev volumes do NOT automatically gain `jackson_user`. This section documents the two supported upgrade paths for operators with an existing `passwd_sso` volume.

> **Pre-condition:** The dev stack is running with the existing volume (`docker compose up` succeeded previously). `PASSWD_JACKSON_PASSWORD` is now present in `.env`.

#### Option (a) — Drop and recreate jackson DB (DESTRUCTIVE — operator-only)

> DESTRUCTIVE — operator-only. All SAML connection configurations stored in the
> `jackson` database are lost. Re-import SAML metadata after the operation.

**Steps:**

```sql
-- Connect as the bootstrap superuser:
-- docker compose exec db psql -U passwd_user

-- 1. Drop the existing database (disconnects any active sessions first)
SELECT pg_terminate_backend(pid)
  FROM pg_stat_activity
 WHERE datname = 'jackson' AND pid <> pg_backend_pid();

DROP DATABASE IF EXISTS jackson;

-- 2. Create jackson_user role
CREATE ROLE jackson_user
  WITH LOGIN NOSUPERUSER NOBYPASSRLS NOCREATEDB NOCREATEROLE
  PASSWORD '<value of PASSWD_JACKSON_PASSWORD from .env>';

-- 3. Recreate the database owned by the new role
CREATE DATABASE jackson OWNER jackson_user;
```

**Expected:**

- No error from any of the three statements.
- `docker compose restart jackson` → Jackson log shows schema creation lines (Jackson recreates its schema on first connect).
- `docker compose logs jackson 2>&1 | tail -20` → no "permission denied" or auth-failure lines.
- SAML connection configs must be re-imported via the Jackson admin UI or API before SSO login can be tested.

#### Option (b) — In-place ownership transfer (data-preserving, DESTRUCTIVE — operator-only)

> DESTRUCTIVE — operator-only. Alters table ownership on the running `jackson` database.
> Run during a maintenance window; no writes to jackson DB during execution.

**Steps:**

```sql
-- Connect as the bootstrap superuser:
-- docker compose exec db psql -U passwd_user

-- 1. Create jackson_user role (if not already present)
DO $$
BEGIN
  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'jackson_user') THEN
    EXECUTE format(
      'CREATE ROLE jackson_user WITH LOGIN NOSUPERUSER NOBYPASSRLS '
      'NOCREATEDB NOCREATEROLE PASSWORD %L',
      current_setting('app.jackson_password', true)
    );
  END IF;
END;
$$;

-- Alternatively, if the DO-block variable binding is unavailable, run psql with
-- the password literal directly:
-- CREATE ROLE jackson_user WITH LOGIN NOSUPERUSER NOBYPASSRLS
--   NOCREATEDB NOCREATEROLE PASSWORD '<value from .env>';

-- 2. Transfer ownership of all objects in the jackson database
-- (run the DO-loop connected to the jackson database)
\c jackson

DO $$
DECLARE
  r RECORD;
BEGIN
  FOR r IN SELECT tablename FROM pg_tables WHERE schemaname = 'public'
  LOOP
    EXECUTE format('ALTER TABLE public.%I OWNER TO jackson_user', r.tablename);
  END LOOP;

  FOR r IN SELECT sequencename FROM pg_sequences WHERE schemaname = 'public'
  LOOP
    EXECUTE format('ALTER SEQUENCE public.%I OWNER TO jackson_user', r.sequencename);
  END LOOP;

  FOR r IN SELECT table_name FROM information_schema.views
            WHERE table_schema = 'public'
  LOOP
    EXECUTE format('ALTER VIEW public.%I OWNER TO jackson_user', r.table_name);
  END LOOP;
END;
$$;

-- 3. Change database owner
ALTER DATABASE jackson OWNER TO jackson_user;

\c passwd_sso
```

**Expected:**

- DO-loop completes without error.
- `\c jackson` succeeds; `\dt` shows tables owned by `jackson_user`.
- `ALTER DATABASE jackson OWNER TO jackson_user` succeeds.
- `docker compose restart jackson` → Jackson connects without error; no "permission denied" lines in logs.
- SAML configs are preserved.

#### Post-migration verification (both options)

```bash
# Confirm jackson_user cannot access passwd_sso tables:
docker compose exec db psql \
  "postgresql://jackson_user:<PASSWD_JACKSON_PASSWORD>@127.0.0.1:5432/jackson" \
  -c "\c passwd_sso"
```

**Expected:** `FATAL: permission denied for database passwd_sso` (or `pg_hba.conf` rejection).

```bash
# Check env-docs still green after .env edit:
npm run check:env-docs
```

**Expected:** Exit 0, no drift warnings.

### Step 3 — HA overlay config render check (C2 / VE2)

**Steps:**

1. `docker compose -f docker-compose.yml -f docker-compose.ha.yml config 2>&1 | grep -A5 "redis:"` (master service section)
2. `docker compose -f docker-compose.yml -f docker-compose.ha.yml config 2>&1 | grep -E "requirepass|masterauth|REDISCLI_AUTH|REDIS_PASSWORD|auth-pass"` (all auth-related lines)
3. `docker compose -f docker-compose.yml -f docker-compose.ha.yml config 2>&1 | grep -A3 "sentinel-1:"` (sentinel entrypoint)

**Expected:**

- Step 1–2: The rendered config contains `--requirepass` on the master (`redis`) service, both replicas (`redis-replica-1`, `redis-replica-2`), and `--masterauth` on all three. `REDISCLI_AUTH` env is present on each Redis service (for healthcheck auth without argv exposure).
- Step 2: Sentinel service sections include `REDIS_PASSWORD:` in their `environment` block (runtime-expansion wiring) and the entrypoint string contains the literal `$REDIS_PASSWORD` (dollar-sign, not the expanded value — runtime expansion, not compose render-time baking).
- Step 3: The entrypoint for sentinel services shows the `chmod 600` step and `exec redis-sentinel` (no dangling `sh` process).

**Optional live check (VE2 — requires HA stack running):**

```bash
docker compose -f docker-compose.yml -f docker-compose.ha.yml \
  exec <sentinel-container> sh -c \
  'ls -l /tmp/sentinel.conf && grep auth-pass /tmp/sentinel.conf'
```

**Expected:** `ls` output shows mode `-rw-------` (600); `grep` output shows `sentinel auth-pass mymaster <actual-password-value>` (real value, not literal `$REDIS_PASSWORD`).

### Step 4 — Dev-flow reachability after C3 (loopback port binding)

**Steps:**

1. `docker compose -f docker-compose.yml -f docker-compose.override.yml config 2>&1 | grep -E '"[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+:[0-9]+:[0-9]+"'`
2. From the **same** machine: `psql "postgresql://passwd_user:<PASSWD_SUPERUSER_PASSWORD>@127.0.0.1:5432/passwd_sso" -c "SELECT 1"` (Prisma Studio uses this path)
3. From the **same** machine: `curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8025`
4. From a **different machine on the LAN** (substitute `<dev-machine-ip>`): attempt `curl --connect-timeout 3 http://<dev-machine-ip>:8025` and `nc -z -w3 <dev-machine-ip> 6379`

**Expected:**

- Step 1: Every port binding in the rendered output begins with `127.0.0.1:` (no `0.0.0.0:` bindings for dev services; the app `3000:3000` stays as-is and is excluded from this check).
- Step 2: Returns `1` — Prisma Studio (`npx prisma studio`) can reach the DB on 127.0.0.1:5432.
- Step 3: HTTP 200 — mailpit web UI is reachable locally.
- Step 4: Both commands time out or report "Connection refused" — the ports are not reachable from another machine on the LAN.

### Step 5 — Locked-out account: UI lockout message (C4)

> **Scope:** This step covers only the UI surface (the message displayed in the browser). The API response shape (status code, `error`, `lockedUntil` fields) is verified by unit tests.

**Pre-condition:** A test account (`<test-user-email>`) with an initialized vault exists and the dev stack is running. The account is NOT currently locked.

**Steps (passphrase path):**

1. Open the vault unlock screen in a browser (e.g. `http://127.0.0.1:3000/<locale>/dashboard`).
2. Enter an incorrect passphrase 5 times in succession.
3. Observe the UI message displayed after the 5th failure.
4. Without waiting for the lockout to expire, attempt to unlock with the **correct** passphrase.
5. Observe the UI message.

**Expected (passphrase path):**

- Step 3: The UI displays the lockout message (English: "Account temporarily locked. Try again in {time}." — the `accountLockedWithTime` key from `messages/en/Vault.json` — or the no-`lockedUntil` fallback "Account temporarily locked due to too many failed attempts.").
- Step 4–5: The correct passphrase also yields the lockout message (not a generic failure or success) — the lockout check fires before any key-material lookup.

**Steps (passkey path):**

1. On the same locked account, open the vault unlock screen.
2. Tap / click "Unlock with passkey" (or equivalent UI control).
3. Complete the passkey ceremony successfully (authenticator prompt).
4. Observe the UI message after the ceremony completes.

**Expected (passkey path):**

- Step 4: The UI displays the lockout message (same `accountLockedWithTime` / `accountLocked` text), not a generic decryption failure — the `/api/vault/unlock/data` endpoint now returns the lockout envelope, and the client parses it into the `ACCOUNT_LOCKED` branch.

## 3. Rollback

| Change class | Rollback procedure |
|---|---|
| `.env` additions (`PASSWD_JACKSON_PASSWORD`, `REDIS_PASSWORD`) | Remove the two lines from `.env`. Run `npm run docker:up` again — Redis will start without a password (insecure); Jackson will fail to connect until the env var is re-added or the DB URL is reverted. |
| Jackson role creation (Option a — drop/recreate) | Restore from a DB volume backup or re-run the original `01-create-jackson-db.sql` as `passwd_user` to recreate the database without a dedicated role. Update `docker-compose.yml` jackson `DB_URL` back to the `passwd_user` form and restart. |
| Jackson ownership transfer (Option b — ALTER TABLE DO-loop) | Run `ALTER TABLE ... OWNER TO passwd_user` for each table, `ALTER DATABASE jackson OWNER TO passwd_user`, and `DROP ROLE jackson_user`. Revert `docker-compose.yml` jackson `DB_URL` and restart. |
| Compose file changes (C2 Redis requirepass, C3 loopback ports) | `git checkout docker-compose.yml docker-compose.ha.yml docker-compose.override.yml docker-compose.logging.yml` to restore the pre-change versions. |
| `infra/redis/sentinel.conf` | File is unchanged by this plan; no rollback needed. |

## 4. Adversarial Scenarios (Tier-2)

### AS-1 — Foreign-name DCR consent claim (C7)

**Setup:**

1. Log in as `<test-user-email>` (User A). Register a DCR client (e.g. via `POST /api/mcp/register`) with `client_name: <client-name>`. Do NOT proceed to consent — leave the client unclaimed (no `tenantId`, no `createdById` claim yet). Confirm the `MCP_DCR_UNCLAIMED_EXPIRY_SEC` (now 1 h) has not elapsed.
2. Log in as `<second-user-email>` (User B) in a separate browser session / incognito window.
3. User B navigates to the OAuth authorization endpoint for the same `client_id` that User A registered.
4. User B submits the consent form to claim the client.

**Expected:**

- The consent route detects that the DCR client was registered by User A (`createdById` mismatch).
- User B receives a consent-error response (HTTP redirect to error page or inline error — the existing consent-error path). The error message uses user-domain wording (not an internal token like `P2002` or `TOCTOU`).
- User A's `MpcClient` row is **NOT** deleted: `GET /api/tenant/mcp-clients` as User A still returns the original record.

### AS-2 — Unauthenticated Redis probe

**Setup:** Dev stack running with `REDIS_PASSWORD` set. Redis port mapped to `127.0.0.1:6379` by `docker-compose.override.yml`.

**Steps:**

```bash
redis-cli -h 127.0.0.1 -p 6379 ping
redis-cli -h 127.0.0.1 -p 6379 keys '*'
```

**Expected:**

- Both commands return `NOAUTH Authentication required.` — the server rejects all unauthenticated commands.
- No data or key names are leaked.

### AS-3 — Locked-account API data fetch with a valid session (C4)

**Setup:** `<test-user-email>` has a valid browser session (authenticated, session cookie present). The vault account is locked (triggered by Step 5 above, or by direct `POST /api/vault/unlock` with wrong passphrase 5 times).

**Steps:**

```bash
# Substitute <session-cookie> with the actual authjs.session-token value
curl -s -X POST http://127.0.0.1:3000/api/vault/unlock/data \
  -H "Cookie: authjs.session-token=<session-cookie>" \
  -H "Content-Type: application/json" \
  -d '{}'
```

**Expected:**

- HTTP 423 (or the same status code returned by `/api/vault/unlock` on the locked branch — must be identical).
- Response body: `{"error":"ACCOUNT_LOCKED","lockedUntil":"<ISO-timestamp>"}` (or equivalent envelope matching `/api/vault/unlock` exactly).
- **No vault key material** (encrypted wrapped key, salt, IV, authTag) appears in the response body — the handler returns before the `prisma.user.findUnique` for key data.

### AS-4 — Magic-link token older than 15 minutes rejected (C5)

**Setup:** Obtain a magic-link URL for `<test-user-email>` (trigger sign-in, intercept the email in mailpit at `http://127.0.0.1:8025`). Wait at least 16 minutes without clicking the link.

**Steps:**

1. After 16+ minutes, open `<magic-link-url>` in a browser.

**Expected:**

- Auth.js rejects the token (expired). The browser is redirected to the sign-in page with an error indicator (e.g. `?error=Verification` in the URL, or an "expired link" UI message).
- The user is **not** signed in.
- A freshly issued magic-link (clicked within 15 minutes) still works normally — regression check.
